# racing_ws 项目审查与改造基线

更新时间：2026-07-14

## 1. 目标与当前结论

用户目标不是单纯把现有固定航点跑通，而是要形成一条可长期复用的比赛链路：

1. 巡线
2. 扫码
3. Nav2 跑航点
4. 返回巡线
5. 全程具备避障能力
6. 本地 `racing_ws` 与车端源码保持同步
7. 出问题时可以通过切回旧 launch 快速回退

当前源码已经具备以下能力：

- 已有巡线检测与巡线控制链路。
- 已有二维码触发 Nav2 固定航点任务链路。
- 已有 `map -> odom_combined` 对齐和二维码矫正能力。
- 已有基于视觉检测结果的近距离锥桶避障能力。

当前源码尚不具备以下能力：

- 巡线过程中自动记录路线并生成导航地图。
- 以“整场任务起点”为唯一全局地图原点的完整录图流程。
- 基于摄像头自动确认当前处于“巡线 / 扫码 / Nav 跑图”哪一条链路。
- 本地到车端的标准化同步部署流程。

结论：当前工程是“静态地图 + 预设航点”的半自动比赛链，而不是“巡线录图 -> 扫码选路 -> 自动跑图 -> 恢复巡线”的完整自动化链。

## 3. 现有节点逻辑总览

### 3.1 巡线链

#### `racing_track_detection`

职责：

- 订阅相机图像。
- 识别赛道中线。
- 发布 `racing_track_center_detection`。
- 监听 `/sign4return`，收到 `5` 时挂起，收到 `6` 时恢复。

结论：

- 它只负责视觉检测，不做任务编排，不做路线记录。

#### `racing_control`

职责：

- 订阅 `racing_track_center_detection`。
- 订阅 `racing_obstacle_detection`。
- 输出 `/cmd_vel`。
- 监听 `/sign4return`，在 `5` 时让出控制权，在 `6` 时恢复巡线。

结论：

- 它已经具备巡线状态机和近距离避障状态机。
- 它不知道二维码语义、地图原点或 Nav2 路线。

### 3.2 扫码链

#### `qr_detector`

职责：

- 监听 `/sign4return`。
- 收到 `0` 后开始订阅图像并执行二维码识别。
- 发布 `qr_code`。
- 把奇偶结果映射为 `/sign_switch=3/4`。
- 发布 `/sign4return=5`，把控制权交给导航链。

结论：

- 当前扫码节点承担的是“识别 + 触发切链”，不是“完整任务状态机”。

### 3.3 导航链

#### `fixed_start_manager`

职责：

- 管理 `map -> odom_combined` 变换。
- 在整场任务起点可向 `/set_pose` 发布归零。
- 在扫码后可根据二维码地标发布地图矫正。

现状：

- `start_pose.yaml` 当前为 `(0, 0, 0)`。
- `map.yaml` 当前分辨率为 `0.01`，即 1 像素 = 1 厘米。
- `map.yaml` 当前原点为 `[0.0, 0.0, 0.0]`。

结论：

- 从地图配置语义上讲，当前地图已经满足“左下角为世界原点、每像素 1 厘米”的基本格式。
- 但实际业务上，二维码矫正点被配置在跑图入口附近，而不是把整场任务的真实起跑点贯穿到整条自动录图链路里。

#### `route_executor`

职责：

- 接收 `qr_code` 和 `/sign_switch`。
- 读取 `routes.yaml`。
- 通过 `/follow_waypoints` 执行预设航点。
- 在导航过程中复用视觉障碍结果做本地避障和恢复。
- 完成后发布 `/sign4return=6`，恢复巡线。

结论：

- 它已经是当前任务编排的核心节点。
- 但它只会执行预设航点，不会自动录点，不会生成地图。

### 3.4 避障链

#### `racing_obstacle_detection`

职责：

- 检测锥桶和停车牌等视觉目标。
- 发布 `racing_obstacle_detection`。

复用方式：

- 巡线阶段由 `racing_control` 使用。
- 导航阶段由 `route_executor` 使用。

结论：

- 当前避障是“视觉局部反应式避障”，不是标准 Nav2 动态障碍层。

## 4. 当前实现与目标之间的差距

### 4.1 地图原点问题

用户要求：

- Nav2 应以所有任务开始点为地图原点，并把该原点视为左下角。

现状：

- `map.yaml` 原点是 `(0, 0, 0)`。
- `start_pose.yaml` 也是 `(0, 0, 0)`。
- 但二维码矫正默认点在 `(2.50, 2.03, 1.57)`，说明当前导航实际上是围绕“二维码入口区域”校正，而不是围绕“整场起点”建立录图基准。

判断：

- 配置格式是对的。
- 业务闭环还没打通。
- 必须补一个“巡线记录 / 地图生成 / 坐标落地”链路，才能真正满足用户要求。

### 4.2 巡线记录问题

用户要求：

- 巡线过程中记录路线。

现状：

- 工程中没有发现 `OccupancyGrid` 生产节点。
- 没有发现巡线阶段的 odom 轨迹记录器。
- 没有发现自动生成 `routes.yaml`、`map.png` 或派生航点的节点。

判断：

- 这是当前工程最大缺口。

### 4.3 任务链路确认问题

用户要求：

- 需要通过摄像头确认当前处在什么链路。
- 巡线特征：地上蓝色虚线。
- 扫码特征：出现二维码。
- Nav 跑图特征：地板为黄色。
- Nav 跑图经过或到达航点时，地上会出现带字白纸。

现状：

- 当前系统没有独立的场景状态确认节点。
- 现有节点默认依靠 `/sign4return`、`/sign_switch` 和二维码结果切链。

判断：

- 需要补一个相机场景守卫节点，对当前链路做视觉确认和异常告警。

## 5. 推荐的增量改造方案

遵循用户要求：优先新建旁路节点，只在 launch 上接入，保留旧链路可回退。

### 5.1 新增节点一：`mission_orchestrator`

职责：

- 成为整场任务的唯一编排入口。
- 显式维护状态机：`LINE_FOLLOW -> QR_SCAN -> NAV_RUNNING -> LINE_RETURN`。
- 管理 `/sign4return`、`qr_code`、`/sign_switch`、导航结果、异常停机。
- 对外提供“当前阶段”和“建议人工放回点”。

不直接替代：

- 不直接重写 `racing_control` 和 `route_executor` 的内部控制逻辑。
- 先把它们当成执行器来调度。

### 5.2 新增节点二：`scene_state_guard`

职责：

- 基于相机图像识别当前链路场景。
- 至少输出四种状态：
  - `line_visible`
  - `qr_visible`
  - `yellow_nav_floor_visible`
  - `waypoint_marker_visible`

用途：

- 给 `mission_orchestrator` 提供“链路确认”和“异常判定”。
- 当状态机与视觉事实不一致时，主动告警并停车。

### 5.3 新增节点三：`track_map_recorder`

职责：

- 在巡线阶段记录 `/odom_combined` 轨迹。
- 以任务起点为唯一世界零点。
- 按 1 像素 = 1 厘米生成内部栅格表示。
- 输出两类结果：
  - 轨迹文件，例如 `recorded_path.yaml`
  - 地图文件，例如 `recorded_map.yaml` + `recorded_map.png`

实现建议：

- 第一阶段先做“轨迹记录 + 航点导出”。
- 第二阶段再做“轨迹转静态像素地图”。

原因：

- 直接一次性做完整 OccupancyGrid 生成风险高。
- 先把“跑图可用的航点”做出来，更快验证主链。

### 5.4 新增 launch：`mission_autorun.launch.py`

职责：

- 作为新方案唯一入口。
- 拉起：
  - 旧巡线检测链
  - 旧巡线控制链
  - 旧二维码链
  - 旧 Nav2 链
  - 新编排节点
  - 新视觉场景守卫
  - 新轨迹记录器

回退方式：

- 若新链路异常，只需切回现有 `race_start.launch.py`。

## 6. 本地与车端同步建议

当前问题：

- 本地是 Git 工作区。
- 车端 `/userdata/racing_ws` 不是 Git 仓库。

因此建议不要把“同步”建立在 Git pull 上，而是建立在“本地构建产物 + 受控部署”上。

建议流程：

1. 本地开发并保存源码基线。
2. 本地生成部署清单。
3. 通过 SSH 把新增包和必要 launch/配置同步到车端。
4. 车端单独构建。
5. 记录每次部署的时间戳和文件列表。

这样做的好处：

- 不会误把车端现场改动覆盖掉。
- 便于回退到上一个稳定版本。

## 7. 测试与人工回退规则

根据用户要求，任何实车测试前都必须先告知；中途发生意外时，必须明确要求把小车放回某一个节点位置。

推荐统一使用以下人工回退点：

1. 停车场起点
2. 扫码点
3. 指定航点白纸点

建议测试顺序：

1. 只测 `scene_state_guard`，不动车。
2. 只测 `track_map_recorder`，巡线但不触发 Nav。
3. 只测扫码到导航切链。
4. 只测 Nav 跑航点与白纸点检测。
5. 最后联调全流程。

强制规则：

- 任一阶段如果状态机判断与摄像头场景矛盾，应立刻停车并请求人工放回最近的回退点。
